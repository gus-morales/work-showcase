# Model Validation Memo: Delinquency Risk Model

**Model owner:** Gustavo Morales, Data Science
**Model type:** Binary classifier (gradient-boosted trees), calibrated, cost-based decision threshold
**Use:** Point-of-sale approve/decline for buy-now-pay-later loan originations
**Status:** Synthetic-data portfolio exercise, written in the structure of an SR 11-7-style validation memo (SR 11-7 is the Federal Reserve/OCC's supervisory guidance on model risk management, the standard reference for how a bank should validate a model before and after deployment; the structure here follows it: data lineage, conceptual soundness, outcomes analysis, ongoing monitoring), not an actual validation of a production system

> All data, figures, and outcomes referenced below come from the synthetic dataset generated by this project (`src/generate_data.py`). No proprietary data, models, or results from any employer are used or implied. This document exists to show the kind of write-up a model like this would need to clear before deployment, not to claim a real model was validated.

## 1. Purpose and scope

This memo evaluates the delinquency risk model documented in the project [README](../README.md) and [notebook](../notebooks/01_delinquency_risk_model.ipynb) against the questions a model risk function typically asks before sign-off: is the data it was built on sound, is the methodology defensible, does it perform as claimed on data it didn't train on, and is there a plan to keep watching it after deployment. It does not cover infrastructure, security, or operational resilience; those are out of scope for this exercise.

## 2. Model overview

At loan application, the model scores the probability that a loan will go 30+ days past due (`delinquent_30dpd`) within the observation window, using applicant and loan-terms features available at origination (bureau score, income, existing obligations, loan amount and term, employment type, tenure on platform, acquisition channel, device type). The raw gradient-boosted output is isotonic-calibrated, then compared against a threshold picked to minimize expected portfolio cost rather than a default 0.5 cutoff (see section 5). `src/serve.py` exposes the trained pipeline as a scoring endpoint; this memo evaluates that same pipeline, not a separate deployment artifact.

## 3. Data lineage and quality

| | |
|---|---|
| Source | Synthetic, generated by `src/generate_data.py` (fixed seed = 42) |
| Volume | 18,092 loans, 9,109 unique customers, 24 months of originations |
| Train / validation / test split | Temporal: months 1-18 train (13,499 loans), 19-21 validation (2,347), 22-24 test (2,246) |
| Missing data | `credit_bureau_score` missing for 9.4% of loans (thin-file applicants: gig/informal workers and recent joiners), handled with a missing-indicator + median imputer fit on the training split only |
| Label | `delinquent_30dpd`, binary, assigned from a hand-specified risk function with idiosyncratic noise (not fit from any real portfolio) |

A temporal split was used instead of a random split because the deployment scenario is inherently temporal: a model is trained on the past and scored against the future, and a random split would let the model implicitly see post-shock information during training that it wouldn't have in production. The test window (months 22-24) includes a simulated macro shock, which is a deliberate stress condition rather than an oversight (see section 6).

Every random draw in the data generator that was added after the original dataset (the fair lending proxy attribute, the survival-analysis time-to-event columns) uses its own independently-seeded random number generator, specifically so that adding those columns couldn't silently perturb the delinquency label or any metric already computed on it. That invariant is enforced by tests in `tests/test_generate_data.py` (`test_bureau_missingness_does_not_change_delinquency_labels`, `test_survival_columns_do_not_change_delinquency_labels`).

## 4. Conceptual soundness

**Methodology.** A gradient-boosted tree ensemble (`HistGradientBoostingClassifier`) was chosen over the logistic regression baseline for its ability to capture interactions (e.g., loan-to-income ratio mattering differently by employment type) without manual feature-crossing. Hyperparameters came from a 5-fold time-series cross-validated randomized search (30 candidates, expanding window, each fold validating on the months immediately after it), not a single train/test split, to reduce the chance of overfitting a threshold or set of hyperparameters to one particular test window.

**Benchmarking against a challenger.** A regularized logistic regression was trained as an interpretable baseline. On the held-out test set, the logistic baseline actually scored a slightly higher single-split AUC (0.801) than the GBM (0.789), though the GBM's cross-validated mean AUC (0.807 ± 0.005 across 5 folds) is higher and less noisy than either single-split number. That gap between a single held-out AUC and the cross-validated mean is exactly the kind of thing this section exists to surface: the held-out test window includes the macro shock, so a single-split comparison is measuring the GBM's AUC on a harder, distribution-shifted window, not a fair apples-to-apples benchmark against the baseline. The GBM was still selected as the production candidate, on the strength of its cross-validated performance and materially better calibration behavior after isotonic correction (Brier score, the mean squared error between predicted probability and actual outcome, lower is better-calibrated: 0.126 vs. 0.172 uncalibrated), not because it won every single-window comparison outright. A validator should treat this as a documented, deliberate choice rather than an oversight.

**Feature leakage.** All feature engineering that requires fitting (missing-value imputation, outlier capping, categorical encoding) is built as a scikit-learn pipeline fit on the training split only and applied unchanged to validation, test, and the monitoring window (`src/features.py`). Ratio features computed without fitting (loan-to-income ratio, installment-to-income ratio) use only information available at origination.

**Assumptions.** The cost assumptions behind the decision threshold (70% loss given default, 6% margin on a good loan) are illustrative, stated explicitly in `src/train.py`, and not fit from real recovery data. Relationships between features and the label are hand-specified with noise (see `src/generate_data.py`), not learned from a real portfolio, since this is a synthetic-data exercise; a real validation would instead assess whether the training data itself reflects the population the model will score.

## 5. Outcomes analysis

| Metric | Value |
|---|---|
| GBM test AUC (single held-out split, incl. shock window) | 0.789 |
| GBM cross-validated AUC (5-fold, expanding window) | 0.807 ± 0.005 |
| Logistic regression baseline AUC (single held-out split) | 0.801 |
| Brier score, raw GBM output | 0.172 |
| Brier score, isotonic-calibrated | 0.126 |
| Cost-optimal decision threshold | 0.05 |
| Expected portfolio cost at optimal threshold vs. naive 0.5 cutoff | 66.7% lower ($158,972 vs. $476,835 on the test set) |
| Precision / recall at the optimal threshold | 28.0% / 91.6% |

The cost-optimal threshold (0.05) is far below the default 0.5 cutoff because the cost asymmetry is large: a missed delinquent loan costs roughly 12x what a wrongly declined good loan costs (70% loss given default vs. 6% forgone margin), so the cost-minimizing policy declines aggressively and accepts a low precision (28%) in exchange for high recall (92% of actual delinquent loans caught). A validator reviewing this in a real setting would want to confirm the underlying loss-given-default and margin assumptions against actual recovery data before accepting a threshold this aggressive, since the entire threshold choice is a direct function of those two numbers.

**Fair lending outcomes** (full detail: README section "Fair lending review"). Approval rate for the synthetic protected-class-proxy group was 37.0% vs. 38.5% for the reference group, a disparate impact ratio of 0.961, comfortably above the 0.80 four-fifths threshold and not statistically distinguishable from noise (p = 0.46, two-proportion z-test, n = 2,246 test-set applicants). Adverse action reason codes were dominated by credit score (54.9% of declines cite it as the primary reason), consistent with credit score being the model's strongest SHAP driver overall.

**Time-to-default outcomes** (full detail: README section "Time-to-default"). A Cox proportional hazards model on the same outcome, reframed as time-to-event, reached a concordance index of 0.814, close to the classification model's own discrimination (AUC 0.789-0.807 depending on split), which is a reasonable cross-check that two different modeling approaches applied to the same underlying risk agree on how separable it is.

## 6. Sensitivity analysis and stress testing

The test window (months 22-24) carries a built-in stress scenario: a simulated macro shock that raises delinquency from 13.5% (months 1-21) to 19.1% in the shocked months. This was a deliberate design choice, not an incidental property of the data, specifically so that model behavior under distribution shift could be assessed rather than assumed. Two findings from that stress test carry directly into the ongoing monitoring plan below:

1. Standard input-drift monitoring (Population Stability Index on every model feature, plus the missing-bureau-score rate) showed no material drift during the shock. Every feature's PSI stayed under the project's 0.20 alert threshold, and the missing-score rate barely moved (9.4% reference vs. 9.9% monitored).
2. Despite that, the model under-predicted the actual default rate during the shock: predicted and observed delinquency rates diverged specifically in the shocked months, a calibration gap that PSI alone never flagged. A rate-mix shift decomposition further ruled out a change in portfolio composition as the explanation: 96.4% of the increase is a within-segment rate effect (existing segments getting riskier), not customers shifting into already-riskier segments, so this is a genuine calibration failure under shock, not a monitoring blind spot about who's applying.

This is the memo's central finding: input-drift monitoring alone would have missed this. Any deployment of this model needs calibration-gap monitoring (predicted vs. observed by segment and time window) running alongside PSI, not instead of it.

## 7. Limitations and known weaknesses

- Synthetic data with hand-specified, mostly-linear relationships (plus one macro shock and one thin-file missingness mechanism). Real portfolios have messier, more interacting effects, label noise, and a more complex missingness mechanism than "thin-file applicants."
- The hyperparameter search covers the GBM's own parameters; it does not test monotonic constraints on features like bureau score, which a real deployment of a risk model would likely require given the model drives approve/decline decisions with compliance exposure.
- Cost assumptions (70% loss given default, 6% margin) are illustrative and stated as such; a real validation would need these benchmarked against actual charge-off and recovery data before the resulting threshold could be relied on.
- The stress-tested window is 3 months. Real ongoing monitoring would track PSI and the predicted-vs-observed calibration gap continuously, not as a one-off before/after comparison.
- The rate-mix shift decomposition is only as informative as the segment dimensions checked (employment type, city tier, merchant category); a mix shift along an unexamined dimension can't be ruled out.
- `demographic_group`, the fair lending proxy attribute, is generated independently of any real demographic data. It demonstrates the disparate-impact and reason-code methodology, not a claim about a real population, and a real fair lending review would use self-reported data or a BISG-style proxy instead.
- The survival-analysis time-to-event columns are generated by clipping a Weibull draw so `event_observed` exactly matches the existing binary label; that consistency is enforced by construction rather than emergent from independent modeling, which a real survival dataset wouldn't need.
- The serving endpoint (`src/serve.py`) is a minimal scoring wrapper: no batching, authentication, model versioning, or canary rollout. It demonstrates that the trained pipeline is servable as-is, not a production deployment design.

## 8. Ongoing monitoring plan

| Check | Cadence | Alert condition |
|---|---|---|
| Population Stability Index, every model feature | Monthly | PSI > 0.20 vs. training reference distribution |
| Missing-bureau-score rate | Monthly | Meaningful deviation from the 9.4% reference rate |
| Predicted vs. observed delinquency rate, by month and by segment | Monthly, with a 60-90 day lag for labels to mature | Gap between predicted and observed rate sustained across at least two consecutive months |
| Rate-mix shift decomposition (mix effect vs. rate effect) | Quarterly, or immediately if the calibration-gap check fires | Material rate effect (existing segments getting riskier), which routes to model retraining rather than underwriting policy changes |
| Fair lending disparate impact and reason-code review | Quarterly | Four-fifths ratio drops below 0.80, or a previously-clean gap becomes statistically significant |
| Full model retrain / recalibration decision | Triggered by any of the above, not calendar-scheduled | Any sustained alert condition, reviewed jointly by model owner and model risk |

The core lesson from section 6 drives the design of this table: PSI alone is necessary but not sufficient, so the calibration-gap check is listed as a first-class monitor rather than a fallback, and the rate-mix decomposition exists specifically to distinguish "the portfolio changed" from "the same portfolio got riskier" so the right fix (policy vs. retrain) gets applied.

## 9. Validation conclusion

For a synthetic-data portfolio exercise, this model would be assessed as follows against the questions a real validation raises: the data lineage is documented and testably clean (section 3), the methodology is standard and benchmarked against a challenger rather than assumed superior (section 4), performance is measured with an appropriate temporal split and honestly reported even where the challenger model wins a comparison (section 5), and the stress test surfaced a real, actionable finding, calibration drift that input-monitoring alone would miss, rather than a clean pass (section 6). The open items before any real deployment would be the loss-given-default and margin assumptions behind the decision threshold (section 5), and the monotonic-constraints question raised in section 7, both of which are conditions a real model risk function would attach to sign-off rather than blockers to the current write-up.
